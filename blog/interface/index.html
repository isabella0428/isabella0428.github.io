Java Interface

April 29, 2019 Yi Lyu

4 minute read

 Twitter

 Facebook

 LinkedIn

 Email

​ I am recently working on java for my summer internship. As a newbie in programming who used to write C++ and python, the term interface confuses me.

​ A formal explanation of interface is that it is actually a collection of methods and fields that the class permits objects of other classes to access. It sounds just like abstract classes in C++ and the required method are just virtual functions ! That was the first thought that came into my mind. I thought I was right until I finally saw abstract class in java… Emmm, anyway. So I did some searches to find out the differences between interface and abstract classes in java.

​ Generally speaking, both interfaces and abstract classes are ways to used for abstraction. Both interfaces and abstract classes are trying to hide the implementation details from the users and provide them a simple “button ” to press on. However, the relationships of these two structures are different. When a subclass extends an abstract class, it is a descendant of the father class. Implementing an interface only means that it fulfills all the required methods in the interface.

​ After comparing them in a general way, let’s look at some specific different usages of these two data structure. There are five main differences listed below1:

  • First a class can implement any number of interfaces but can only subclass at most one abstract class

​ Interfaces are more like some specified tasks to be done and “smart” classes can accomplish multiple easy tasks at the same time. But as a subclass of abstract class, it is more like the son of the father class. You can’t have multiple fathers at the same time :D

  • An abstract class can have nonabstract methods; but all the methods of an interfaces are effectively abstract.

​ To better illustrate the idea, i wrote the following example codes.

​ Suppose that it’s dinner time and we want to cook ourselves a dinner and construct an abstract class cookDinner. Since a balanced dinner is good for the health, we define cookMeat and cookDinner as abstract methods that must be realized in our subclasses. However, we can have various ways to cookMeat and cookVegetable, we should all go to the kitchen first, thus we don’t need to make method goToKitchen a unique function for all subclasses.Instead, we define a common method in the abstract class.

  abstract class cookDinner{
      void goToKitchen(){
          ...
      }
      abstract void cookMeat();
      abstract void cookVegetable();
  }

​ When Alice wants to cook her own dinner, she should have a unique class with her own unique methods to do these things. However, she must realize methods cookMeat() and cookVegetable(), so that AliceCookDinner can be a kind of cookDinner.

  class AliceCookDinner extends cookDinner{
      void cookMeat(){
          //Alice's style of cooking meat
      }
      
      void cookVegetable(){
          //Alice's style of cooking vegetable
      }
  }

​ But what if it is cook that needs to cookDinner? It becomes the job that he should accomplish. He should be able to cook meat and vegetable in many different ways and accomplish many orders. An order is exactly an interface. So a good cook can implement multiple interfaces.

  interface Order1{
      public void cookMeatA();	//cook meat in method A
      public void cookVegetableB();	//cook vegetable in method B
  }
  
  interface Order2{
      public void cookMeatC();	//cook meat in method C
      public void cookVegetableD();	//cook vegetable in method D
  }
  
  class greatCook implements Order1, Order2{
      public void cookMeatA();
      public void cookMeatC();
      public void cookVegetableB();
      public void cookVegetableD();
  }
  • An abstract class can declare and use fields; an interface cannot, although it can create static final constants

In order to be used for all classes, the methods defined in interfaces should be public. Since interfaces cannot be instantialized, all the constants should be static and final in order to prevent being altered by its implementation methods.

  • An abstract class can define constructors; an interface cannot

Since an interface cannot have constructors, it cannot be instantialized.

To sum up, both interfaces and abstract classes are used for abstraction. If we only need abstract methods, then we can just use abstract classes to replace interfaces. Interfaces are stricter ways to do abstraction and can form n-tier architecture which is essential in software engineering.

  1. Design Patterns in Java, Appendix A, Page 347, Steven John Metsker, William C. Wake, 2007 [return]
